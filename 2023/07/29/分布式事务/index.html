数据不一致解决思路

数据不一致的根因有两种

1.下游业务超时，导致业务状态不稳定，此时上游业务回滚/异常，但下游业务实际成功。

2.同一业务流程中有多个下游服务的RPC调用，前序业务执行成功，后续业务调用失败，前序业务未回滚。

针对以上两种情况，业内有比较通用的解决方案，具体如下：

1.对于接口超时导致的未知错误，需要通过重试来明确执行结果，因此对下游服务均保证接口幂等（技术上所有业务接口都可以通过幂等表/幂等键来保证接口幂等）。

2.通过补偿型事务进行向前重试或者向后回滚，以达到数据的最终一致性。业务常用方案如TCC、Saga等。

补偿型事务选型

方案名称 描述 适合的场景 优点 缺点
1 最大努力通知 顾名思义，即本地事务成功后不断地对下游业务进行同步/异步的正向重试补偿，以此达到数据最终一致。 + 下游业务场景通过重试一定会成功
+ 可靠性要求不高的非核心链路，如数据异步更新、消息通知等。因为极端情况下可能会丢失事务信息。		 + 实现简单，对业务侵入性小 + 事务上下文信息坑会丢失；丢失后业务手动补偿难度较打；
+ 只支持正向重试补偿
2 可靠消息型事务 使用消息事件表存储需要补偿的业务信息，通过本地事务保证业务和消息事件一起成功失败。本地事务成功后通过RPC或者MQ的方式调用下游服务，若调用成功删除本地事件；若调用失败通过定时任务扫描事件表，查询出未执行成功事件，对下游业务进行重试。 + 下游业务场景通过重试一定会成功
+ 对可靠性要求较高的场景。		 + 通过本地事务保证了强一致性；
+ 通过记录的事件信息可以方便的进行事务恢复和补偿。		 + 对业务有侵入，需要在DB更新写入事件信息。
+ 只支持正向重试补偿。
3 TCC TCC(Try-Confirm-Cancel)实际上是服务化的两阶段提交协议，业务开发者需要实现者三个服务接口，第一阶段服务由业务代码编排来调用Try接口进行资源预留，所有参与者的Try接口都成功了，事务管理器会提交事务，并调用每个参与者的Confirm接口真正提交业务操作，否则调用每个参与者的Cancel接口回滚事务。 + 对事务隔离性要求高的服务，Try阶段的存在可以很好地保证TCC事务直接彼此的隔离性
+ 业务可以方便地抽象出TCC的三个阶段。尤其适合新服务，没有老逻辑的包袱。		 + 一致性、隔离性好 + 对业务而言需要抽象出2个阶段，有一个业务改造成本
4 Saga 业务处理由一系列sub-transation Ti（字事务，也叫分支事务）组成，每个TI都有对应的补偿动作Ci，补偿动作用于撤销Ti造成的结果。某一步进行失败后，可以又失败情况决定向前重试还是向后回滚。向后回滚为撤销掉之前所有成功的sub-transation，使得整个执行结果撤销；向前重试为不断地正向重试。 + 当执行失败的参与者资源状态会对其他服务或者其他参与者产生副作用，就可以考虑向后回滚的方式。
+ 没有预留动作，易于改造兼容老的业务逻辑。
+ 补偿服务即正向服务的“反向”，易于理解，易于实现。		 + 易用改造实现，业务上理解成本较低 + 不保证隔离性，如果出现了并发请求的情况，可能会导致无法回滚，脏读等情况。

对于以上4中方案实现，都需要下游支持接口的幂等

最大努力通知和可靠消息事务必须保证正向重试一定能够成功

TCC和Saga必须保证逆向回滚一定能够成功。

资源的定义

资源的定义：会引起数据变更的操作，就认为是当前服务要操作的资源，主要包括：

  • 下游业务的所有写接口
  • DB、ES等存储类系统的所有写操作
资源特征 名词解释说明 对应的供应链资源 分析
1 资源约束型 在执行业务时，资源方会校验资源是否充足。 资源充足即成功，不够即失败。 库存锁库、出库； 账户余额的冻结、扣款； 促销资源预占； 需要考虑事务隔离。若事务不隔离会存在脏读/回滚失败问题。 ·例1：正向是加资源的流程，回滚逆向会扣减资源，会存在并发情况导 致资源已被其他事务扣减，就会存在资源不足导致回滚失败的情况。 •失败后无法重入，需要考虑链路上资源的回滚。
2 业务校验型 在执行业务时，会进行业务校验。如单据状态、会计期、物 品启停用、APM校验、DB乐观锁校验等等。 业务校验通过后业务一定成功。 备注：供应链现状正常的校验都会由上游业务前置进行，所 以除开BUG因素理论上业务上无校验型资源。 - • 若存在此类场景，业务校验失败后无法重入，需要考虑链路上资源的回 滚。 ·业务校验型无需关注事物隔离，后续与资源约束型一同讨论。
3 DB资源 业务提交本地事务更新DB数据，也是当前业务的资源之 一。 备注：特指有乐观锁校验的DB资源，单DB创建/查询属于不 限制型。 - •业务中存在的乐观锁在业务并发执行时可能会导致DB更新失败。失败后， 需要考虑链路上资源的回滚。 •非二阶段业务，DB资源提交后，回滚难度较大。
4 不限制型 下游业务最终一定会成功。 注意状态是最终一定成功，可能会出现业务阻塞导致的中间 失败，但最终可以成功也认为是不限制型资源。 如虚拟出入库，可能受成本重算影响失败，但是成本重算结 束后再次重试就会成功。 ES资源； 虚拟出入库； 库存释放预占、入库、反 审核； 账户余额的解冻、退款； 下游资源的创建 • 适用正向补偿，最终一定会成功。
5 总结：
1. 不限制型资源可以正向补偿，4种补偿事务实现都能支持
2. 资源约束型、业务校验型和DB资源都需要考虑链路上的资源回滚，只能考虑TCC和Saga
3. 资源约束型需要考虑事务隔离，最佳选择为TCC、Saga型事务在该场景下需要业务支持。

业务分析&归类

对供应链现有的业务依赖场景进行梳理汇总，如下图所示：

整体看，供应链业务资源主要分为了2大类，分别为不限制类资源和全局限定类资源。

1.不限制类资源，主要涉及单据的创建、更新和消息推送等操作。不限制类资源的特点是业务上操作均会成功。

2.全局限定类资源，主要涉及促销、账户和库存，全局限定类资源的特点是会被上游多入业务方操作；有资源限定、超出限定业务失败。详细资源操作如下：

  • 促销：主要涉及促销资源的预占，若促销资源不够则预占失败。
  • 账户：主要涉及账户的冻结、解冻、扣款和退款。对于冻结、扣款这类资源减少操作，金额不够业务失败。对于解冻、退款等资源增加操作，业务上一定成功。
  • 库存：主要涉及库存的预占、锁库和出入库。对于锁库、出库等资源减少操作，库存不足业务失败；对于入库等库存增加操作，可能因为成本重算等业务校验不过导致失败，但最终业务上一定会成功。

1.限定资源仅包括对促销、账户和库存本身的余额/结存数据。资源操作也是对以上数据的操作。

2.每个业务会进行的非资源校验如物品启停用、仓库是否可用、管辖权限/范围、会计期、盘点等统称为业务上下文。

3.业务上下文同样会导致业务的失败，目前必须采用预校验方式，由上游业务提前进行预先校验，后续仅是对资源的操作。目前供应链都是使用了预校验方式，会在业务侧进行所有业务上下文的校验。

综上梳理，供应链业务资源特征整体可分为以下三类：

  • 资源类型：定义资源是不限制类资源还是全局限定类资源。不限制资源业务上操作一定会成功；全局限定资源业务上操作可能会失败，并且全局资源是多个业务方共享的资源，因此对全局

限定类资源操作需要考虑资源隔离。

  • 资源操作是增加还是减少：站在资源方的视角，资源操作会增加资源还是减少资源。资源的减少需要考虑资源是否充足，会出现失败的情况，此时即使重入也会失败。资源的增加一般业务

    上不会存在限制，最终一定成功。资源增加/减少只需要全局限定类资源关注，不限制资源无需张注。

  • 业务方是否关注资源操作结果：业务方包含了调用方和客户端，指业务上是否要使用资源操作的结果。最直接的表现是，客户端是否需要展示资源操作结果或者调用方是否需要操作结果继

    续进行后续业务，如：订货单分单生成配送单场景，需要将生成的配送单展示给用户。对于全局限定资源减少操作，业务方一定需要及时感知结果；对于全局限定资源增加、不限制类资源

的操作，业务方可能不关注结果。

5.3 典型场景推导

结合单列出来的本地事务资源操作，总共有6类资源操作：【全局限定资源减少】、【业务方关注的全局限定资源增加】、【业务方不关注的全局限定资源增加】、【业务方关注的不限制资源】、【业务方不关注的不限制资源】、【本地事务操作】。

极端场景的处理方案

接口超时处理

处理原则如下：

  1. 选择正向重试补偿时，若资源方接口超时，直接进行重试补偿，如果重试多次均不成功，则人工介入处理。
  2. 选择 TCC / Saga 事务时，资源方接口超时，RPC 层面直接重试3次，如果成功则继续；如果重试失败，则进行回滚 当前资源 + 前序资源。

空回滚处理

空回滚的定义：事务没有经过正向阶段就执行了 cancel 方法。下游业务需要支持空回滚成功，且 cancel 后不允许进行正向提交。

空回滚出现的场景举例如下：

  1. 选择 TCC 事务，Try 阶段有资源锁定失败，需要对 Try 阶段资源进行回滚，可能存在资源方还未调用 try 方法，但是会调用回滚方法。
  2. 选择 TCC / Saga 事务，正向接口一直超时，资源方未完成处理；此时业务方发起回滚，可能存在回滚请求先于正向请求进行处理。
  3. 选择正向重试补偿时，对下游业务进行异步补偿，可能出现异步补偿一直失败（如库存此时在进行成本重算），此时用户发起了单据的撤销“回滚”。

空回滚带来的副作用是事务悬挂问题，如下：

  1. 事务空回滚后，此时正向请求到达资源方，资源方进行了正向资源的扣减，但此时实际上事务已经进行回滚。

治理方案：

  1. 需要新增事务控制表，记录事务执行状态。
  2. 正向流程到达时，若事务控制表记录为空，插入一条记录状态为初始化。后续回滚请求到达时发现事务控制表里有记录，状态为初始化，继续执行回滚。
  3. 如果回滚请求到达时发现事务控制表里没有自身 XID 的记录，说明自己是一个空回滚，需要新增一条记录，用于防止后续的正向事务悬挂。
  4. 如果正向流程到达，发现事务控制表记录不为空，意识到自己是一个悬挂的方法，因此忽略执行。

业务上下文变化处理

业务上下文的定义：上下游共性的非资源校验，如物品启停用、管辖权限、会计期等。

供应链业务对于 界限上下文 目前均是由业务方进行前置的校验，极端情况可能出现上游校验通过，下游因为业务界限上下文变化出现失败。

  1. 选择 TCC / Saga 事务，理论上不会存在上下文变化情况，因为每个接口仅重试3次，再次失败就会回滚，时间在毫秒级。
  2. 选择正向重试补偿时，可能出现正向补偿一直失败，需要人工介入，但是后续人工介入重试发生了上下文变化，但是无法重入。如异步进行库存入库，此时库存存在进行成本重算导致一直重试失败，后续人工介入时，出现物品停用等校验不过的情况。

治理方案：

  1. 同物品启停用，之前和产品对齐有个原则为只在订货时校验，后续业务不再校验。但该方案无法适配所有场景，如会计期校验。
  2. 对于选择正向重试补偿的场景，若重试时出现业务上下文校验失败的情况，人工介入处理。下游资源方增加参数，可跳过业务校验。

事务方案对比选型

从上文分析看，针对不同的场景事务方案对比点在于：最大努力通知和可靠消息型事务、TCC 和 Saga 型事务之间。

Swan 对四种事务类型都进行了实现，还提供了异常报警、手动补偿、大盘监控等基建能力，并且注解式接入方案也较为轻量。因此基于 Swan 实现方案进行对比选项。

最大努力通知和可靠消息型事务对比

Swan 对于最大努力通知型事务和可靠消息型事务的实现比较类似，主要差异点在于最大努力通知型事务由于事务日志的插入和本地提交不在同一事物中，极端情况下可能因为插入事务日志失败，导致事务丢失。简单对比如下：

Swan 实现会通过 DB/Cellar 存储事务日志，因此在管理平台上可查询异常事务信息，对异常事务进行重试等操作，解决了事务全局失败后不好手动补偿的问题，并可限定 set。 2. 支持同步、异步、同步几次后转异步等多种重试机制，也可配置按执行异常进行重试/不重试 3. 提供了异常通知、报警等基础基建，全局事务失败后进行报警；业务上可实现全局回调方法，在全局事务成功后或失败后回调执行。 4. 若选择 DB 存储事务日志，需要业务方建相关表，业务上无需感知事件写入逻辑。 适配度高 能够满足场景1的业务诉求 复杂度低 无需下游业务提供回滚接口，只需要保证正向接口的幂等 稳定性中 极端情况下事务日志插入 DB 失败，导致事务日志数据丢失 可扩展性高 只调整现有流程编排 开发/维护成本低 只需要下游业务支持接口幂等和空回滚

从供应链业务上看 通知型事务 > 消息型事务

  1. 虽然通知型事务可能丢失事务日志，但是 DB insert 失败的场景过于极端。
  2. 消息型事务由于事务日志插入 DB 是在本地事务中，目前 Swan 的实现是基于代理 Spring 的声明式事务来隐式织入，保证在同一事务。但供应链业务中目前都是使用的 template 型事务，改造有一定成本。

方案对比：

1. 隔离性对比

因为 Saga 型事务不保证隔离性，可能导致回滚时【全局限定资源扣减】的失败。此时若选择 Saga 型事务需要业务层面改造支持，一般来看需要业务做如下处理：

  • 预校验：最大概率保证正向提交时无资源不足问题。
    • 举例如下：发货出库时先查询出库存当前库存量，在校验阶段预先校验，最大程度的减少提交时资源不足的情况。
  • 业务在回滚时不限制资源。
    • 举例如下：配送退货会给用户退款，极端情况下退款金额被用户订货时使用，配送送货回滚进行扣款时发现余额不足。所以业务上需要在回滚时可选择将账户扣为负。
    • 结合供应链自身业务场景，扣款、出库操作存在不能扣为负的强限制场景（如赠送余额不能扣为负、库存不能为负、库存开关关闭）。预占、冻结操作业务上能够允许为负。

隔离性问题出现前提：

  1. 业务存在并发调用场景，若不存在并发，无隔离性问题。
  2. 正向是资源增加，回滚时资源扣减场景，正向资源增加后并发扣减，导致回滚时资源不够回滚失败。对于正向是资源扣减的场景，回滚是资源增加，也无需关注隔离性问题。

2. 业务适配度对比

  • 如果业务上能够兼容，如库存预占、账户冻结：TCC ≈ Saga
  • 如果不考虑业务兼容或者业务上存在不能兼容的情况，如库存出库、账户扣减：TCC ≥ Saga

3. 开发/维护成本对比

  • 业务方改造成本：TCC 和 Saga 都需要业务发进行编排修改和组件接入，相对来说 TCC 涉及到业务 2 阶段的适配，而 Saga 的正向流程与业务更贴合，所以业务方接入成本上 TCC 最高，Saga ≥ TCC。
  • 资源方改造成本：TCC 和 Saga 都需要资源方需要提供如回滚能力、幂等能力、空回滚处理。相对来说 TCC 会有一个资源方二阶段改造成本。供应链已知的【全局限定资源】中，库存已支持锁库，业务可按二阶段接入。资金账户已完成二阶段改造的设计对详，详见：账户记账二阶段改造 & 冻结金额拆分，预计 15PD 改造成本。促销资源的改造成本和资金账户类似。因此都接入 TCC 预计的成本会多 30PD。Saga ≥ TCC。
  • 维护成本：TCC 的二阶段有一定的理解成本，问题排查上查看多张表记录，维护成本相对较高。Saga ≥ TCC。

结论：

  1. 除部分特殊场景外均使用 Saga。
  2. 特殊场景：退货收货、差异处理/撤销场景使用 TCC。

决策如下：

  1. Saga 事务落地和维护成本更低。
  2. 目前预占、冻结操作业务层面支持“扣减为负”的情况。
  3. 从目前供应链业务数据看，平均每个门店每天单单量 <1，有并发冲突导致出现隔离性问题的概率较低。
  4. 正向流程为资源扣减的场景，逆向为资源增加，业务上无需考虑隔离性导致的回滚失败问题。

综上，绝大部分场景选择 Saga 事务能够解决问题，且落地成本和风险可控，收益上大于选择 TCC。

退货收货、差异处理/撤销场景正向流程为【全局限定资源增加】，逆向流程为【全局限定资源减少】，且业务上存在不能为负的限制（数据修复成本较高，且可能有资损），因此选择接入 TCC，保证业务限制不能突破。

方案选型结论

说明

  1. 方案选型只涉及到当前业务，长链条上的业务按当前思路各自套用。
  2. 下游业务必须进行的改造
    a. 接口必须支持幂等
    b. 若提供回滚/撤销接口，下游业务必须支持空回滚
  3. 【业务方不关注的不限制资源】和【业务方不关注的全局限定资源增加】在【本地事务】后执行，进行正向重试补偿，使用 最大努力通知型事务。
  4. 【全局限定资源减少】、【业务方关注的全局限定资源增加】和【业务方关注的不限制资源】在【本地事务】前执行，需要考虑资源的回滚，使用 Saga 型事务。

幂等方案

方案思考

幂等方案主要需要从3个方面进行考虑：

  1. 如何判断是重复请求
  2. 如何进行并发控制
  3. 重复请求如何处理
考虑维度 方案说明
1 1. 如何判断是重复请求 判断是否重复请求主要看业务上能否有唯一标识，因此主要有以下3种场景和方案： - 场景1：无业务唯一标识的新增 此场景下需要服务端颁发 token，通过 token 判断是否重复，做防重复提交 - 场景2：有业务唯一标识的新增 此场景下可以使用业务唯一键判断是否重复 - 场景3：有业务唯一标识的更新 此场景下可以使用业务数据对应的唯一键 + 状态/版本号 判断是否重复
2 2. 如何进行并发控制 - 使用数据库 - 唯一索引 - 悲观锁：select for update - 乐观锁：update set XXX and version = 2 where version = 1 - 使用分布式锁
3 3. 重复请求如何处理 - 场景1：消息重复消费 此场景下，不需要给上游一个返回，重复执行只需要丢弃 - 场景2：前端重复提交 此场景下，需要给用户一个提示，那么重复请求可以直接抛出一个 DuplicateRequestException 异常，由上层捕获，前端展示“请不要重复请求” - 场景3：底层的转账接口 此场景下，同一笔转账，重复调用第一次若成功了，第二次应该返回相同“成功”，若第一次失败了，第二次返回业务执行结果
4 4. 总结说明 1. 如何判断是重复请求：供应链是单据驱动的业务，天然可以使用单据号相关信息作为业务唯一键，无需考虑 token 颁发机制。 2. 如何进行并发控制：由于供应链业务存在唯一键，因此直接使用 DB 唯一索引或者分布式锁即可。 3. 重复请求如何处理：在供应链场景下对于重复请求均返回成功，根据不同场景，有不同的结束方式，应该由业务方按场景自己决定是丢弃、抛异常、直接返回，还是执行业务逻辑，后续不再赘述。

6.2 方案选型

6.2.1 并发控制选型

按上述分析，使用 DB 唯一键或者分布式锁来进行并发控制，因此可有如下方案：

方案简介 业务适配度 复杂度 性能 可扩展性 开发/维护成本
1. 业务表中增加幂等键，幂等键作为唯一索引 主要流程如下： - 新增单据时在业务表中写入唯一键 - 收到请求时通过业务唯一键查询业务表，若有记录按返回值直接返回 - 并发情况下，上游的重复请求 DB 会抛出 DuplicateKey 异常，直接返回成功 适配度中 只能满足新增时的幂等，更新操作无法保证 复杂度中 涉及到的表需要增加幂等字段 性能高 无需引入额外多操作 可扩展性中 后续涉及到的幂等操作都必须要在表中新增字段 开发/维护成本中 业务表中需要维护业务唯一键信息
2. 新增幂等表，幂等表中新增业务主键为唯一索引，幂等表记录与本地事务一起提交 主要流程如下： - 本地事务提交时一起写幂等表 - 收到请求时通过业务唯一键查询幂等表，若有记录按返回值直接返回 - 并发情况下，上游的重复请求 DB 会抛出 DuplicateKey 异常，直接返回成功 适配度高 所有幂等场景都能支持 复杂度高 需要引入幂等表 性能中 有部分的 DB 操作 可扩展性高 后续幂等场景均在幂等表中实现即可 开发/维护成本中 写幂等表和本地事务一起提交，会导致代码的可维护性下降
3. 新增幂等表，幂等表中新增业务主键为唯一索引，幂等表记录先于本地事务提交 主要流程如下： - 收到请求时即插入幂等表，若抛出 DuplicateKey 异常，按返回值直接返回 - 如果后续业务流程有失败的情况需要回滚幂等表 适配度高 所有幂等场景都能支持 复杂度高 需要引入幂等表 性能中 有部分的 DB 操作 可扩展性高 后续幂等场景均在幂等表中实现即可 开发/维护成本低 幂等流程单独维护
4. 使用分布式锁，按业务唯一键进行加锁 主要流程如下： - 收到请求时加分布式锁 - 加锁完成后通过业务唯一键查询业务表，若有记录按返回值直接返回 适配度低 只适合业务本身有唯一键的业务，若依赖上游唯一键则不适用 复杂度中 代码层面加锁即可 性能中 有额外的加锁操作 可扩展性高 对业务操作没有影响 开发/维护成本低 只需要加锁即可

总结说明：

  1. 所有方案都需要可能涉及到前序资源的回滚，由分布式数据一致性方案解决。
  2. 方案一业务适配度较低且后续可维护性和扩展性较低，暂不考虑该方案。
  3. 方案二和方案三的差异点在于幂等表的写操作时机，相较来看方案二优于方案三，因为方案三极端情况还需要回滚幂等表。
  4. 方案四缺陷在于需要依赖业务本身的唯一键，若唯一键由上游传入则不适用，因为需要根据唯一键来查询是否执行。

综上：

  1. 优先选择方案二，若存在业务本身的唯一键可选择方案四。

6.2.2 重复请求判定选型

按上述分析，供应链业务可使用单据号相关信息作为业务唯一键，因此可有如下方案：

方案简介 业务适配度
1. 使用单据号作为业务唯一键 适配度低 只能满足 1 对 1 的新增场景，1 对多新增和更新操作无法满足 如： - 配送收货单多次收货入库，此时按收货单单据号作为唯一键，会导致后续部分收货无法创建入库单 - 前端的发起的单据更新，对一张单据的多次更新，也会导致直接触发幂等，更新失效
2. 使用单据号 + 版本号作为业务唯一键 适配度高 所有场景都能支持，版本号变更一定是有 DB 操作，因此单据号 + 版本号能够唯一的确认一次写操作流程。
3. 总结说明： 使用 单据号 + 版本号 作为业务唯一键

6.3.2 选型结论

  • 使用 单据号 + 版本号 作为业务唯一键
  • 优先选择 幂等表 来进行并发控制，幂等表记录先于本地事务提交；若存在业务本身的唯一键可选择分布式锁来进行并发控制

注意： 下游业务在处理上游业务发起的回滚请求时，除了要回滚资源，还需要在幂等表中删除对应的幂等键。
